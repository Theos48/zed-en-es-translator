# Contract: Operational Provider Selection

## Configuration matrix

The existing four environment keys remain the complete configuration surface.
No real secret may be stored as a value of these keys.

| Mode | `TRANSLATOR_PROVIDER` | `TRANSLATOR_PROVIDER_URL` | `TRANSLATOR_PROVIDER_API_KEY_ENV` | `TRANSLATOR_ALLOW_REMOTE_PROVIDER` |
|---|---|---|---|---|
| Mock default | absent or `mock` | absent | absent | absent/false |
| LibreTranslate local | `libretranslate` | `http://127.0.0.1:5000` | absent | absent/false |
| Azure remote | `azure_translator` | absent | required safe variable name | exactly `true` |

Unknown, duplicate, incomplete, conflicting, empty, or mode-inapplicable values
fail with `PROVIDER_NOT_CONFIGURED` before provider contact.

## Locality

- Mock: `offline`
- Reviewed LibreTranslate profile: `local`
- Azure Translator: `remote - confirmation required`

CLI and LSP execution plus the Zed code-action label derive from the same
parsed `ProviderConfiguration`. Startup fails closed if locality and selected
provider cannot be produced from one configuration instance.

MCP/Agent Panel remains a compatibility bridge rather than an F011 product or
acceptance surface. It continues to consume the shared core selection boundary
and its existing regression suites must pass, but this feature adds no new MCP
provider flow or real-service validation matrix.

The UI never includes provider name, URL, model, image, path, account, tier, or
credential-reference details in the locality label.

## Target validation

### LibreTranslate

- Scheme: `http`.
- Host: exact IPv4 loopback `127.0.0.1`.
- Port: exact `5000` for the supported operational profile.
- Base path, query, fragment, userinfo, alternate hostname, IPv6, DNS name,
  redirect, inherited proxy, and API key are forbidden.

The existing generic LibreTranslate adapter may retain broader loopback test
support internally, but the F011 operational commands and acceptance profile
use only the exact target above.

### Azure Translator

- Scheme: exact `https`.
- Host: exact `api.cognitive.microsofttranslator.com`.
- Port: default HTTPS only.
- Path: exact `/translate`.
- Query: exact `api-version=3.0`, `from=en`, `to=es` generated by the adapter.
- A configured URL, custom region/endpoint, userinfo, fragment, redirect,
  inherited proxy, or caller-provided query is forbidden.

## Request gates

The existing request/segment/response contracts remain authoritative. Every
real invocation uses this order:

1. input/file/format/limit validation;
2. segmentation and protection;
3. provider configuration and target validation;
4. fresh remote confirmation when locality is remote;
5. secret blocking when locality is remote;
6. provider contact;
7. response/cardinality/output-limit validation;
8. reconstruction without mutation.

Confirmation applies to one invocation only. A denial, dismissal, stale or
mismatched confirmation response, or prior confirmation returns
`REMOTE_CONFIRMATION_REQUIRED` before network contact. If the bound document
changes or closes while confirmation is pending, target validation instead
returns `INVALID_INPUT` before network contact.

## Error mapping

| Condition | Existing normalized code |
|---|---|
| Missing/invalid configuration or key reference | `PROVIDER_NOT_CONFIGURED` |
| Remote consent absent/invalid | `REMOTE_CONFIRMATION_REQUIRED` |
| Secret detected before remote contact | `SECRET_DETECTED` |
| Client timeout or provider HTTP 408 | `PROVIDER_TIMEOUT` |
| DNS/TLS/auth/quota/rate limit/other HTTP failure | `PROVIDER_FAILED` |
| Empty/malformed/non-text/cardinality mismatch/oversized response | `PROVIDER_FAILED` |

Messages are generic and redacted. There is no automatic retry.
