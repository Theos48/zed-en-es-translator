# Matriz de decisiones

Este documento registra decisiones estables de producto, arquitectura y proceso.
La implementacion detallada de la feature activa vive en `specs/<feature>/`.

## Estado de decisiones

| D002 | Producto | La arquitectura debe dejar espacio para mas idiomas en el futuro. | Aceptada | Evita acoplar contratos internos a un unico par de idiomas. |
| D003 | UX | El MVP tecnico muestra la traduccion en el Agent Panel de Zed. | Matizada por D065 | Fue la via documentada mas cercana para validar herramientas externas en Zed, pero no es la experiencia final de producto. |
| D004 | UX | El MVP no modifica ni reemplaza texto en el buffer. | Aceptada | El objetivo inicial es lectura asistida, no edicion automatica. |

| D022 | Salida | Modo normal limpio, con errores claros solo cuando algo impida traducir. | Aceptada | Mantiene lectura simple sin ocultar fallos accionables. |
| D023 | Spec Kit | Spec Kit queda inicializado y las features formales viven bajo `specs/<feature>/`. | Aceptada | La implementacion activa necesita una fuente operativa unica para requisitos, plan, contratos, tareas y quickstart. |

| D026 | Documentacion | Usar la skill `mermaid-diagrams` para Mermaid fuente dentro del repo hasta que se aprueben validadores o exportadores. | Aceptada | Es mas acotada para nuestro caso y mejora diagramas sin instalar herramientas, generar imagenes ni cambiar politicas del sistema. |
| D027 | Integracion TS-Rust | El servidor MCP TypeScript llamara primero a un binario CLI Rust. | Reemplazada por D056 | En F005 el servidor MCP sera Rust y llamara a `translator-core` directamente. |
| D028 | Pruebas | Usar `make test` para Rust via contenedor, `vitest` para TypeScript cuando llegue MCP, y tests de integracion contra el CLI Rust. | Reemplazada por D057 | F005 mantiene pruebas Rust via contenedor y no introduce TypeScript/Vitest. |
| D029 | Estructura | Usar `zed-extension/`, `crates/translator-core/`, `crates/translator-cli/`, `mcp-server/`, `tests/` y reservar `specs/` para Spec Kit. | Reemplazada parcialmente por D058 | F005 ubica el servidor MCP en `crates/translator-mcp/`; wrapper Zed queda para F006. |
| D030 | Contrato core | `TranslateRequest` tendra campos comunes de idioma, tono, preservacion e `input_kind`; aceptara exactamente una entrada publica: `source_text` para texto directo o `workspace_root` + `file_path` para archivo validado. | Aceptada | El MCP necesita schema explicito para tools; nuestro core necesita distinguir texto directo de lectura segura de archivo sin mezclar configuracion del proveedor ni aceptar texto de archivo suministrado por el caller. |
| D031 | Contrato core | La salida publica sera `translated_text` en exito y `code` + `message` en error. | Aceptada | MCP soporta contenido de texto y errores de ejecucion con `isError`; nuestro UX pide salida limpia salvo fallos accionables. |
| D032 | Errores | Usar codigos iniciales `INVALID_INPUT`, `UNSUPPORTED_LANGUAGE_PAIR`, `UNSUPPORTED_FILE_TYPE`, `FILE_TOO_LARGE`, `FILE_NOT_FOUND`, `PROVIDER_NOT_CONFIGURED`, `REMOTE_CONFIRMATION_REQUIRED`, `PROVIDER_FAILED`, `INTERNAL_ERROR`. | Aceptada | Cubre validacion MCP, limites de archivo, privacidad remota y fallos de proveedor definidos en nuestras decisiones previas. |
| D033 | Provider | El `Provider` recibira segmentos traducibles y devolvera segmentos traducidos; el core protege codigo/formato y reconstruye la salida. | Aceptada | Separa responsabilidad de traduccion del analisis de Markdown/codigo y permite mocks deterministas para TDD. |
| D034 | Archivos | `translate_file` aceptara inicialmente `.md`, `.markdown`, `.txt`, `.rs`, `.ts`, `.tsx`, `.js`, `.jsx`, `.py`, `.sh`, `.bash` y `.zsh`. | Aceptada | Cubre documentacion simple y lenguajes de codigo prioritarios con comentarios/docstrings definidos; formatos como `.rst`, `.adoc` y `.mdx` quedan fuera hasta tener preservacion especifica. |
| D035 | Archivos | `translate_text` aceptara texto seleccionado o pegado sin validar extension de archivo. | Aceptada | La seleccion ya acota el riesgo y permite leer fragmentos de cualquier archivo sin habilitar traduccion completa de formatos no soportados. |
| D036 | Comentarios | En codigo, traducir solo comentarios/docstrings reconocidos por reglas del lenguaje y conservar el resto del codigo intacto en la salida. | Aceptada | Cumple el objetivo de lectura sin traducir codigo y mantiene contexto alrededor de comentarios/docstrings. |
| D037 | Spec Kit | El primer ciclo formal de Spec Kit sera un MVP tecnico: core Rust, `MockProvider`, contrato CLI, limites y pruebas negativas de seguridad. | Aceptada | No mezcla contrato/base segura con proveedor real; permite TDD sin red ni secretos. |
| D038 | Archivos | En el primer ciclo Spec Kit, `translate_file` aceptara solo `.md`, `.markdown` y `.txt`; soporte de codigo completo queda gated por segmentador confiable y pruebas. | Aceptada | Evita prometer preservacion de codigo antes de tener parser/segmentador robusto; D034 queda como objetivo del MVP usable posterior. |
| D039 | Seguridad de archivos | `translate_file` solo podra leer dentro del workspace autorizado por Zed, tras canonicalizacion. | Aceptada | Reduce riesgo de exfiltracion por paths arbitrarios desde una tool MCP. |
| D040 | Seguridad de archivos | Rechazar paths absolutos no autorizados, traversal con `..`, symlinks que escapen del workspace, archivos ocultos sensibles, contenido binario y entradas no UTF-8. | Aceptada | La validacion por extension no basta para proteger secretos o archivos fuera de alcance. |
| D041 | Limites | El limite de archivo completo y `translate_text` sera 20 KiB UTF-8 por defecto; cada segmento tendra maximo 4 KiB, maximo 256 segmentos, salida maxima 40 KiB y timeout inicial de proveedor de 15 s. | Aceptada | Fija limites verificables para TDD, evita entradas enormes y acota memoria/latencia. |
| D042 | Privacidad remota | Los proveedores remotos estaran deshabilitados por defecto; toda llamada remota requiere configuracion explicita y confirmacion por cada traduccion, aplicada por servidor/core. | Aceptada | La seguridad no debe depender solo de la UI o del cliente MCP; el servidor debe negar por defecto. |
| D043 | Privacidad remota | El `file_path` no se enviara a proveedores remotos en el MVP; el proveedor solo recibira segmentos traducibles. | Aceptada | Minimiza metadata sensible y conserva privacidad del workspace. |
| D044 | Logs | Los logs nunca incluiran `source_text`, segmentos, traduccion completa, secretos, headers, tokens ni rutas sensibles sin redaccion. | Aceptada | Evita filtraciones por logs de Zed, MCP, CLI o proveedor. |
| D045 | Configuracion | No se versionaran `.env` reales; el MCP/CLI usara entorno minimo por allowlist cuando la plataforma de lanzamiento permita limpiar el entorno heredado. | Matizada por D064 | Reduce exposicion accidental de secretos y mantiene el host/proyecto limpio; Zed context-server stdio no permite limpiar el entorno heredado desde `zed_extension_api` 0.7.0. |
| D046 | Contrato CLI | El CLI Rust permanece como frontera publica testeable del core. | Aceptada | Conserva una frontera estable para integraciones futuras; los detalles operativos viven en `specs/<feature>/`. |
| D047 | Sandbox Provider | Proveedores CLI/HTTP futuros usaran allowlist; CLI sin shell, args estructurados, cwd controlado y env minimo; HTTP local solo loopback, remoto con HTTPS y sin redirects peligrosos. | Aceptada | Reduce ejecucion arbitraria, SSRF y fuga de entorno cuando se agreguen proveedores reales. |
| D048 | Formato | `preserve_formatting` sera obligatorio en el MVP, aunque exista en el contrato como campo explicito. | Aceptada | No habra modo de romper formato/codigo hasta que existan pruebas que lo justifiquen. |
| D049 | Errores | Agregar codigos `PATH_NOT_ALLOWED`, `NON_UTF8_INPUT`, `NO_TRANSLATABLE_SEGMENTS`, `SECRET_DETECTED` y `PROVIDER_TIMEOUT`. | Aceptada | Cubre controles nuevos de archivo, privacidad, ausencia de segmentos y timeouts. |
| D050 | Secretos | Si se detectan patrones obvios de secreto antes de una llamada remota, la operacion se bloqueara o exigira confirmacion reforzada; en el MVP tecnico se bloqueara. | Aceptada | Preservar codigo no evita que comentarios o Markdown contengan credenciales. |
| D051 | Supply chain | Rust y TypeScript usaran lockfiles, versiones revisadas y auditoria antes de publicar o habilitar proveedores reales. | Aceptada | Controla riesgo de dependencias, scripts de instalacion y binarios externos. |
| D052 | Agent Panel | La seleccion de Zed se tratara como UX no garantizada hasta validarla manualmente; el flujo base aceptado es pegar texto o pasar path permitido al Agent Panel. | Matizada por D072 | Evito depender de una capacidad no comprobada de Zed para el primer ciclo, pero no autoriza nuevas features de producto basadas en Agent Panel. |
| D053 | Documentacion | La constitucion manda sobre principios, `specs/<feature>/` manda sobre la feature activa, y `docs/` registra decisiones estables, investigacion y roadmap. | Aceptada | Evita mantener dos planes vivos y reduce deriva documental conforme avance la implementacion. |
| D054 | Toolchain Rust | Rust se ejecuta por defecto mediante la imagen Docker oficial fijada en el `Makefile`; no se instala `rustc`/`cargo` globalmente para este proyecto. | Aceptada | Cumple la politica del host y deja el flujo reproducible para TDD. |
| D055 | Spec Kit | `docs/feature-map.md` alimenta futuras iteraciones `specify -> clarify -> plan -> tasks -> implement`, pero el detalle operativo se mueve a `specs/<feature>/` cuando una feature se formaliza. | Aceptada | Conserva el detalle util para futuras specs sin duplicar la fuente de verdad de la feature activa. |
| D056 | MCP | F005 implementara el servidor MCP en Rust usando `rmcp` y llamando a `translator-core` directamente. | Aceptada | El SDK Rust oficial ya cubre tools/stdout-stdio, evita Node global/proyecto y reduce fronteras de privacidad. |
| D057 | Pruebas | F005 usara pruebas Rust de contrato/integracion MCP dentro del contenedor del proyecto; no se agregara Vitest en esta iteracion. | Aceptada | Mantiene el host limpio y valida el servidor MCP en el mismo workspace Rust. |
| D058 | Estructura | Agregar `crates/translator-mcp/` como binario de servidor MCP; no crear `mcp-server/` TypeScript en F005. | Aceptada | El servidor es una pieza Rust de producto, mientras el wrapper Zed sigue separado para F006. |
| D059 | Transporte MCP | F005 soportara solo MCP por stdio; HTTP/Streamable HTTP queda fuera. | Aceptada | Stdio encaja con lanzamiento por comando desde Zed y evita abrir superficie de red. |
| D060 | Zed wrapper | F006 usara `zed-extension/` con `extension.toml`, Rust/WASM mediante `zed_extension_api = "0.7.0"` y `context_server_command` para lanzar el binario local `translator-mcp` por ruta configurada, sin shell, descargas, red ni entorno arbitrario agregado por el wrapper. | Matizada por D064 y D072 | Alinea la integracion con la API vigente de Zed, conserva el servidor MCP existente como fuente de verdad y mantiene la politica de host limpio; la validacion manual en Zed depende de prerrequisitos propios de Zed y no instala toolchains globales desde esta feature. |
| D061 | Arquitectura wrapper | Para F006 se elige wrapper Rust/WASM en lugar de un wrapper mas fino fuera de Rust porque la API de extensiones de Zed para `context_server_command` ya usa `zed_extension_api`, permite devolver un `zed::Command` estructurado sin shell y mantiene el control de argumentos, entorno y redaccion dentro del mismo lenguaje y workflow del proyecto. | Matizada por D072 | Reduce fronteras tecnicas nuevas y valida integracion con reglas de seguridad, pruebas y host limpio; no decide que el producto futuro deba vivir en Agent Panel. |
| D062 | Calidad Rust | Fijar `[workspace.lints.rust]`/`[workspace.lints.clippy]` en el `Cargo.toml` raiz (heredado por `translator-core`, `translator-cli` y `translator-mcp` via `[lints] workspace = true`) y una tabla `[lints]` equivalente en `zed-extension/Cargo.toml`, denegando `clippy::all` y el estilo no estandar. | Aceptada | Aplica la skill `rust-best-practices` (cap. 2.5): fija a nivel de `Cargo.toml` la misma politica que ya exigia `make clippy` con `-D warnings`, para que `cargo build`/`cargo check`/`cargo clippy` fuera del `Makefile` no pierdan esa red de seguridad; verificado sin regresiones con `make test`, `make fmt`, `make clippy` y `make test-zed-extension`. |
| D063 | Zed wrapper | El objetivo original de F006 de que un artifact faltante/no usable sea visible en <=15s se reemplaza por "visible dentro de la ventana de inicializacion del context server de Zed" (~60s observado), en vez de seguir intentando un fast-fail dentro del sandbox WASM. | Aceptada | Se revisó el codigo fuente de `zed_extension_api` 0.7.0 y se confirmo que no expone ninguna primitiva viable para validar una ruta de host absoluta fuera del worktree desde WASM: `std::fs::metadata` no es confiable sin preopen WASI, y `zed::process::Command::output()` reproduce el mismo timeout de la modal ya observado con el preflight `/usr/bin/test`. Es una limitacion de plataforma, no un bug de implementacion; re-evaluar si una futura version de `zed_extension_api` expone una verificacion rapida de ruta o un timeout de inicializacion configurable. |
| D064 | Zed wrapper | En F006, la garantia de entorno minimo se limita a que el wrapper no lee ni reenvia su propio entorno heredado y solo agrega `RUST_LOG` allowlisted; el proceso real `translator-mcp` aun hereda el entorno completo del proceso Zed por limitacion de plataforma. | Aceptada | Se confirmo en `crates/context_server/src/transport/stdio_transport.rs` de Zed que el transporte stdio crea `std::process::Command`, llama `.envs(binary.env)` y no llama `.env_clear()`. Como `std::process::Command` hereda el entorno del padre por defecto y `zed_extension_api` 0.7.0 no expone control para limpiarlo, el wrapper no puede garantizar aislamiento end-to-end del entorno. El riesgo queda acotado porque `translator-mcp` no enumera ni registra variables arbitrarias de entorno y la extension sigue rechazando `extra_env`, proveedores remotos y secretos configurables. |
| D065 | UX final | El Agent Panel es un puente de validacion para usar las tools MCP existentes, no la experiencia final del producto. La experiencia final debe ser una accion propia de la extension de Zed, invocable desde menu contextual, comando o boton, que permita traducir texto seleccionado o contenido del documento abierto y mostrar la traduccion sin requerir configuracion de Agent en Zed. | Aceptada | El usuario final no deberia necesitar configurar perfiles Agent ni depender de un modelo intermediario para una accion explicita de traduccion. Esta decision conserva el objetivo de privacidad: proveedor local o gratuito/no pago, remoto solo con consentimiento explicito, y no-mutacion automatica del buffer. |
| D066 | Producto | La meta de producto es una extension nativa de traduccion para Zed, disenada como producto propio para trabajo tecnico: preview, preservacion de formato, control explicito para copiar/insertar/aplicar, proveedores locales o gratuitos/no pago, y privacidad por defecto. | Aceptada | Evita que el roadmap se conforme con un wrapper MCP o un prompt de Agent. La propuesta de valor esta en una UX integrada y segura para documentacion, Markdown, comentarios y fragmentos tecnicos, con no-mutacion por defecto y proveedor intercambiable. |
| D067 | Proveedores | F004/F005 selecciona como primer proveedor real un servicio local/self-hosted compatible con LibreTranslate, manteniendo `MockProvider` como default y remoto deshabilitado por defecto. | Aceptada | Permite obtener traduccion real sin convertir un proveedor remoto, pago o con API key en requisito base; el detalle operativo vive en `specs/005-real-provider-config/`. |
| D068 | Contrato proveedor | La configuracion de proveedor vive fuera del texto a traducir, como configuracion controlada del proceso/servidor y de la extension Zed; la confirmacion remota por solicitud se modela como dato aditivo con default `false`, sin enviar URL, secretos, headers ni workspace metadata al proveedor. | Aceptada | Preserva compatibilidad de solicitudes existentes, evita mezclar secretos/configuracion con contenido traducible y permite que servidor/core apliquen default-deny antes de cualquier contacto remoto. |
| D069 | Tecnologia proveedor | El adaptador real se implementara detras del trait Rust `Provider` en `translator-core`, con cliente HTTP bloqueante minimo (`ureq` 3.x previsto) configurado con timeout explicito y sin uso de proxies heredados. | Aceptada | Mantiene CLI y MCP sobre la misma frontera testeable, evita introducir async en el core sincronico actual y reduce riesgo de fugas por entorno heredado o logging no redaccionado. |
| D070 | Spec Kit | Cada iteracion debe reportar explicitamente el estado de `speckit-checklist`, `speckit-analyze` y `speckit-converge` como ejecutado, no aplicable o bloqueado por prerequisito; no se deben omitir silenciosamente gates posteriores aunque la sesion se detenga antes de implementacion. | Aceptada | Evita cerrar flujos parciales como si fueran completos y deja evidencia concreta del comando/prerrequisito que justifica saltar una fase. |
| D071 | Roadmap | F010, flujo directo de extension Zed sin Agent, debe promoverse antes de F009/publicacion. | Aceptada | La meta de producto es una extension nativa usable sin configurar Agent; publicar antes consolidaria una experiencia puente como si fuera final. |
| D072 | Roadmap | Desde F006, cualquier feature que toque Zed debe orientarse a una extension propia y directa; Agent Panel solo puede usarse como puente tecnico, compatibilidad o fallback justificado por limitaciones concretas de la API de Zed. | Aceptada | Evita construir varias features alrededor de Agent Panel para migrarlas al final. F006 es la fundacion de la extension, no una decision de convertir Agent en la UX principal. |
